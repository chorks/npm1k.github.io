npm1K are we licensed yet?

npm, the awesome source code package manager for JavaScript and stuff, recently updated its guidelines on license metadata in package.json files. The new guidelines ask that package maintainers use the SPDX standard’s license expression syntax to indicate how their work is licensed in a machine-readable way. You will get a warning if you don’t.

In most cases, following the new guidelines is as simple as making sure the license property (not licenses, as for RubyGems) is a string containing one of the license identifiers on SPDX' official list. That means Apache-2.0 instead of Apache License, version 2, GPL-3.0 instead of GPLv3, and BSD-2-Clause or BSD-3-Clause, &c., instead of ambiguous BSD. Fortunately, many of the most popular options are the most straightforward: MIT and ISC are both fine. You can even Unlicense or WTFPL, you rebel, you.

Who cares? Nobody cares! … until they have to do a license check. Then, thanks to how amazing npm is, you’re left staring down a nigh on bottomless well of node_modules directories to trawl, manually, for LICENSE files and hints in README. All the reckless joy of npm install --save-dev comes back to haunt you. But shouldn’t some lousy program do that work for you? Of course it should! But those programs need good, clean data to chew on. There are tons of licenses, and there is no way everybody is going to decide how to format them … or ever where to put them in a package. Predictable metadata is the answer.

The following is a list of packages among the 1,000 most-depended-upon (the “npm1K”) that aren't yet part of that better world. There are handy links for making quick pull requests to fix package.json if you’d like to help them out. Given how vital these packages are, each one that properly codes its licensing makes it that much easier to get npm and JavaScript adopted in new places, many of which are dipping timid corporate toes into open-source for the very first time.
